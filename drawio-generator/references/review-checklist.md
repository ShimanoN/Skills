# Design Review Checklist

Use this checklist for final review of drawio flowcharts.
Report each file as `✅ No issues` or `⚠️ Issues found`.

---

## A. Logic Flow Correctness

| # | Check |
|---|-------|
| A-1 | Input nodes (`INP_xxx`) → process steps → output nodes (`O_xxx`) match the design intent |
| A-2 | Yes/No branch condition labels are logically consistent with their direction (right column vs. downward) |
| A-3 | Output node colors (ok=green / alarm=red / ng=pink / idle=light-blue) match the state's meaning |
| A-4 | Loop-back and RESET paths are fully represented — no missing paths |

---

## B. Readability

| # | Check |
|---|-------|
| B-5 | Each block label is concise and descriptive (not too long, not too vague) |
| B-6 | Yes/No edge color-coding (bold green/red) is applied to the correct edges |
| B-7 | Top-pin entry (Γ-shape) arrows clearly indicate the target node |
| B-8 | `[requires confirmation TX]` annotations are recognizable as open action items |

---

## C. Specification Completeness

| # | Check |
|---|-------|
| C-9 | All execution paths of each FB are covered: normal / fault / timeout / RESET |
| C-10 | Output node labels accurately describe the state returned to the caller |
| C-11 | Both EMO (NC contact DI-001) and RESET (DI-003) input paths merge correctly (FB_Alarm) |
| C-12 | Solenoid valve combinations (SOV-01/EV-OUT/EV-BYP) are covered for each STATE (FB_ValveControl) |

---

## D. Design Concerns

| # | Check |
|---|-------|
| D-13 | Blocks tagged `[requires confirmation]` have their implementation concerns documented |
| D-14 | In polling FB designs, blocks re-executed every cycle (e.g., buzzer) are reviewed for side-effects |
| D-15 | Fan-in points (multiple paths converging on one output) are free of mutual-exclusion problems |

---

## Common Issue Patterns (from past reviews)

### 🔴 High Priority

| Pattern | Description | Fix |
|---------|-------------|-----|
| **Double count increment** | MAIN's `count++` block and FB_DataLog's `count++` both execute in the same cycle | Centralize `count` management — remove one of them |
| **Missing DI read block** | An `IfBlock` references a DI value without a preceding `RtxDIReadBlock` | Insert a read block immediately before the decision node |

### 🟠 Medium Priority

| Pattern | Description | Fix |
|---------|-------------|-----|
| **No input edge to EMO state** | No edge in the diagram leads to `STATE_ALARM (E)` | Use `alarm.from_all: true` + annotation edge (auto-generated by script) |
| **Two edges from a process node** | A `VariableAddBlock` or similar directly emits two Yes/No edges | Insert an `IfBlock` decision node in between |
| **Double KVL acquisition** | FB receives `INP_KVL` but also runs `KeyValueListCopyBlock` internally | Delete the internal B1 and connect `INP_KVL → B2` directly |

### 🟡 Low Priority

| Pattern | Description | Fix |
|---------|-------------|-----|
| **Loop variable not initialized** | `i = 1` or similar init is absent from `steps` | Add initialization step |
| **Precondition block missing tag** | A block requiring implementation confirmation has no tag | Add `[requires confirmation TX]` tag |
| **Buzzer re-trigger every cycle** | A block in EMO state executes every cycle causing repeated side-effects | Tag with `[requires CKD confirmation T2]` |

---

## Output Format

Report results in this structure:

```
### [filename]
✅ No issues  — or —  ⚠️ Issues found

(if issues exist)
- **[A/B/C/D]-[N]** NodeID/EdgeID: Issue description → Recommended action
```

Close with a **Summary** and **Follow-up Action Items** list.

### Priority Legend

| Symbol | Priority | Criteria |
|--------|----------|----------|
| 🔴 | High | Directly causes implementation bugs, numeric errors, or safety hazards |
| 🟠 | Medium | Divergence from implementation, incomplete specification, severely reduced readability |
| 🟡 | Low | Inconsistent notation, optional improvements, label additions |
